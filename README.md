# ЛК Интерра

Мобильный личный кабинет абонента интернет-провайдера **Интерра** (`interra.ru`,
г. Первоуральск). Работает поверх биллинга NetUP UTM5: показывает веб-кабинет,
подтягивает баланс нативно, шлёт push-уведомления и даёт полезные инструменты
(диагностика сети, «кто в моём Wi-Fi», виджеты баланса, Siri, Apple Watch).

Приложение — Flutter (один код на iOS и Android). Бэкенд рассылок — Node.js.

| `app/`    | Мобильное приложение (Flutter, iOS + Android) — **основной продукт**           |

| `.github/`| CI (GitHub Actions): анализ, тесты, сборка релизного APK как артефакт          |

## Как всё связано

```
┌─────────────────────────────┐   регистрация push-токена    ┌───────────────────────────┐
│  Приложение (app/)           │ ───────────────────────────▶ │  Бэкенд (server/)          │
│  • кабинет в WebView         │                              │  • база устройств (SQLite) │
│  • нативный баланс           │ ◀──── push (FCM → APNs/FCM)  │  • рассылка (firebase-admin)│
│  • виджеты / Siri / часы     │                              │  • панель оператора        │
│  • диагностика / сеть Wi-Fi  │                              │                            │
└─────────────────────────────┘                              └───────────────────────────┘
        │  прямые запросы к биллингу (HTTPS + пиннинг корней Let's Encrypt)
        ▼
   stat.interra.ru/cgi-bin/utm5   (UTM5: вход по SMS-токену, страница кабинета, баланс)
```

Три независимых канала:

1. **Приложение ↔ биллинг UTM5** — вход и данные кабинета (напрямую, без нашего сервера).
2. **Приложение → бэкенд** — только регистрация устройства для push (кто, какой токен, какие категории).
3. **Бэкенд → приложение** — push-уведомления через Firebase Cloud Messaging.

## Суть работы

- **Вход** — по номеру телефона и SMS-коду через штатный CGI-эндпоинт `bbb`
  биллинга UTM5. Приложение один раз регистрируется и хранит выданный 24-значный
  токен в защищённом хранилище (Keychain / EncryptedSharedPreferences), а по нему
  при каждом открытии получает свежую ссылку на кабинет (живёт ~30 минут).
- **Кабинет** показывается в WebView (`stat.interra.ru`); поверх страницы UTM5
  вживляются фирменные стили (в т.ч. тёмная тема), а баланс и номер лицевого счёта
  дополнительно **парсятся** со страницы и выводятся нативно — в шапке, виджетах,
  на часах и через Siri. Последний снимок кабинета кэшируется для офлайна.
- **Безопасность** — опциональный замок (Face ID / отпечаток и/или код-пароль) с
  настраиваемой задержкой, заслонка приватности в переключателе задач,
  сертификат-пиннинг для запросов к своим хостам.
- **Инструменты** — пошаговая диагностика сети (где именно проблема) и сканер
  «Устройства в моём Wi-Fi».
- **Уведомления** — через Firebase Cloud Messaging: оператор из панели бэкенда
  рассылает их всем, сегменту (по категориям-тегам) или конкретному логину.

Подробнее по частям — в [`app/README.md`](app/README.md) и
[`server/README.md`](server/README.md).

### Что нужно сделать при заливке в сторы

- **Android (Google Play):** релизный ключ уже есть (`app/android/key.properties`
  + `upload-keystore.p12`, оба вне git). Не потерять ключ/пароли или включить Play
  App Signing. См. [`docs/STORE_CHECKLIST.md`](docs/STORE_CHECKLIST.md).
- **iOS (App Store):** при активации платного Apple Developer аккаунта включить
  capability Push и поставить `aps-environment = production` (сейчас намеренно
  отключено). См. [`docs/IOS_PUSH_SETUP.md`](docs/IOS_PUSH_SETUP.md).
- Полный список — [`docs/APP_STORE_READINESS.md`](docs/APP_STORE_READINESS.md).

## Быстрый старт

```bash
# приложение
cd app && flutter pub get && flutter run          # запуск на устройстве/эмуляторе
flutter analyze lib test && flutter test          # проверки
flutter build apk --release                        # Android-сборка (нужен Android SDK)
flutter build ios  --release                       # iOS-сборка (нужен macOS + Xcode)

# бэкенд
cd server && npm install && npm start              # без ключа Firebase — режим dry-run
```
